\subsection{Imports von Modulen}

\begin{frame}[fragile]{Pakete und Module einbinden}
	\begin{itemize}
	\item Das gesamte Python-\tb{Ökosystem} baut auf dem Importieren (fremder) Pakete auf.
	\pause
	\item In Python gibt es \tb{zwei Möglichkeiten}, Pakete einzubinden.
	\pause
	\item[]	\begin{lstlisting}[language=pytiny]
import random
from random import randint
    \end{lstlisting}

	\item Erstere Struktur importiert das \tb{gesamte Modul} und stellt \tb{seinen Namespace} objekt-ähnlich zur Verfügung.
	\footnote{Verändert der eigene Code den Wert einer globalen Variable des Moduls, so ändert er sich überall, wo das Modul mit der \texttt{import}-Struktur importiert ist.}
	
    \item Die \pp{from}-Struktur bindet eine \tb{Referenz} des importierten Objekts an den \tb{gegebenen Namen} innerhalb des importierenden Namespaces.
	
	\item Das Einbinden von fremder Funktionalität via \lpl{from module import *} \pause
	widerspricht dem Konzept der Namespaceunterteilung und soll nicht verwendet werden.
	\end{itemize}
	
\end{frame}


\begin{frame}{Ein Modul}
	\begin{itemize}
		\item Jede Python-Datei ist ein eigenständiges \pp{Modul} mit eigenem \tb{Namespace}.
		\pause
		\item \tb{Mehrere} Dateien in einem Ordner können zu einem gemeinsamen \pp{Package}\footnote{\texttt{Packages} werden oft auch in Pythons eigenen Docs als \texttt{Module} bezeichnet. Package ist im Grunde eine spezifischere Bezeichnung für ein Modul, welche etwas über die Struktur verrät. Zudem gilt immer: Ein Package besteht aus ein- bis n-Modulen.} zusammengefasst werden, welches dann als \tb{eins} importiert werden kann.
		\pause
		\item[] Hierzu muss in dem Verzeichnis eine Datei \pp{__init__.py}
		\footnote{Einer der ganz wenigen Orte, wo \texttt{from x import *} Import sinnvoll sein kann, um alle Elemente anderer Package-Dateien aus init heraus zur Verfügung zu stellen.} liegen.
		\pause
		
		\item Jedes Modul hat einen \tb{Namen}. Dieser besteht aus \pp{packagename.modulname}.
		\pause
		\item Einzige \tb{Ausnahme} ist die initial ausgeführte Datei. Diese heißt immer \pp{__main__}\footnote{Das Main-Modul eines durch \texttt{multiprocessing} erzeugten Prozesses heißt \texttt{\_\_mp\_main\_\_}}.
	\end{itemize}
\end{frame}


\begin{frame}[fragile]{Ausführung und Import eines Moduls}
	\begin{itemize}
        \item Der \tb{Import} eines Moduls sorgt für dessen \tb{komplette Ausführung}.
        \pause
        \item Hierbei sollten lediglich \tb{Funktionen und Konstanten} definiert werden.
        \pause
		\item Wird ein Modul hingegen explizit als \tb{Programm aufgerufen}, so soll dieses nicht nur Funktionalitäten definieren, sondern auch eine \tb{Funktionalität ausführen}.
		\pause
		\item Aus diesem Grund soll ein Modul in der Lage sein zu \tb{unterscheiden}, ob es der initial ausgeführte Entrypoint, oder ein Import ist.
		\pause
		\item Dies geht über die \tb{Überprüfung} der in \pp{__name__} verfügbaren Information.
		\pause
	\end{itemize}
    \begin{lstlisting}[language=pytiny]
# some imports and functions

if __name__ == '__main__':
    some_functionality()
            \end{lstlisting}
\end{frame}



\begin{frame}{Ein nützlicher Trick beim direkten Ausführen von Modulen I}
    
	\begin{itemize}
		\item Es gibt Situationen, in denen ein \tb{Package direkt ausgeführt} wird, statt von anderem Code importiert zu werden. Beispiel: Command-Line Programme, wie \textsl{youtube-dl}\furl{https://github.com/ytdl-org/youtube-dl}.
		\pause
		\item Wird ein Package von einem \tb{anderen Skript} importiert, wird es zuerst \tb{vollständig initialisiert} und ist dann normal verfügar.
		\pause
		\item[] Ist ein Package initialisiert, so können Module dieses Packages \tb{innerhalb} des Packages durch \pp{eigener_package_name.modul} adressiert werden.
		\pause
		\item Wird ein Package \tb{direkt ausgefürt}, ist dieses \tb{nicht initialisiert} und importierte Module im selben Packages werden \tb{nicht} mehr durch die \pp{package.modul}-Adressierung gefunden.
	\end{itemize}
\end{frame}


\begin{frame}[fragile]{Ein nützlicher Trick beim direkten Ausführen von Modulen II}

Eine simple, aber effektive \tb{Lösung} ist die Verwendung dieses Codeblocks, \tb{bevor} die eigenen Imports in der initial-ausgeführten Datei des Packages angegeben werden.
\pause


\begin{lstlisting}[language=pytiny]
# make sure we can import other modules from our package when it is run directly
# package is None when the module is executed directly (and not imported)
# a module is frozen when it's packaged in an executable that doesn't rely on external dependencies
if __package__ is None and not hasattr(sys, "frozen"):
    import os.path

    # get path of the current file
    path = os.path.realpath(os.path.abspath(__file__))
    # add current directory to list of directories interpreter searches when a module is required
    sys.path.insert(0, os.path.dirname(os.path.dirname(path)))

# import things from your package here!
from packaging_sample_project import some_functions
\end{lstlisting}
	
\end{frame}

	\begin{itemize}
		\item In der \lpl{.toml} gibt es ein Feld namens \lpl{dependencies}, dort können Dependencies angegeben werden.

		\begin{itemize}
			\item Versionsnummern können mit \lpl{==}, \lpl{>=},  \lpl{\~=} spezifiziert werden.
			\item Hierbei bedeutet \lpl{\~=} \glqq kompatibel mit\grqq
			\item Selbige Art von \tb{Versionpinning} kann auch in der \lpl{requirements.txt} vorgenommen werden.
		\end{itemize}
	\end{itemize}
	
	
	\begin{lstlisting}[language=pytiny]
dependencies = [
    fastapi ~= 0.92.0,  # größer gleich 0.92.0, aber kleiner 0.93.x
    openai-whisper >= 20230124,
]
	\end{lstlisting}

	\end{lstlisting}
	
	
\begin{itemize}

\section{Typehinting II}
\subsection{Abstrakt}
\begin{frame}{Warum Typehinting}
	\begin{itemize}
		% TODO Spiderman meme von variable auf str auf int
		\item Python ist \tb{dynamisch} typisiert. Eine Variable kann alles sein.
		\item Für mehr \tb{Nachvollziehbarkeit} des Codes wurde Typehinting eingeführt.\footnote{PEP484: \url{https://peps.python.org/pep-0484/}, eingeführt mit python3.5}
		\item Typehints geben den \tb{erwarteten} Typ bzw. die \tb{möglichen Typen} einer Variable an.
		\item[] Dieses hilft
		\begin{itemize}
			\item beim \tb{Dokumentieren} des eigenen Codes für sich selbst und für andere.
			\item der \tb{IDE} dabei, den Code zu verstehen und sinnvollere \tb{Vorschläge} zu machen.
			\item Libraries dabei, \tb{automatische} Checks und Conversions vorzunehmen.
			\item statischen \tb{Typecheckern}, wie \pp{mypy}\furl{https://mypy-lang.org/}, dabei, unseren Code zu testen.
		\end{itemize}
	\end{itemize}	
\end{frame}


\begin{frame}{Was machen Typehints intern?}
	\begin{itemize}

		
		% TODO: soll dieser Punkt rein und ist der korrekt so? - nicht hier. und gilt nur partiell fir zb Iterable
		% \item An sich kommunizieren TypeHints nur, welche Special Methods und andere Funktionaitäten ein Objekt zur Verfügung stellt.
		
		\item Typehints werden zur \tb{Laufzeit nicht überprüft}, sind noch verfügbar und sind z.B. mit dem \pp{inspect}\furl{https://docs.python.org/3/library/inspect.html\#introspecting-callables-with-the-signature-object}-Module oder \pp{obj.__annotations__} \tb{auslesbar}.
		\pause
		
		\item Einige \tb{Libraries} machen aktiv davon Gebrauch, um Signaturen zu erkennen oder Datenschemata zu erstellen und zu \tb{validieren}.
		\pause

	\end{itemize}

\begin{frame}[fragile]{Weiterführendes Typisieren}
	\begin{itemize}
		
		\item \tb{Typen} können auch in \tb{Variablen} (\pp{Type aliases}) gespeichert werden.
		\footnote{Es gibt keine Naming Convention für Aliases. Eine verbreitete Notation ist \texttt{typenameT}. Pythons Docs unterscheiden nicht zwischen Alias und \glqq normaler\grqq~ Variable.}
		\pause
		
		\item[]\begin{lstlisting}[language=pytiny]
numberT = int | float
x: numberT = 3.14
		\end{lstlisting}
		\pause
		
		\item \tb{Weitere} Typen können mit \pp{TypeVar}\footnote{Ebenfalls aus dem \texttt{typing} Modul} erstellt werden.
		
		\item[] Dies ist nützlich, um z.B. \tb{generische} Typen \pp{TypeVar("T")} zu erstellen.
		
		\item[]\begin{lstlisting}[language=pytiny]
K, V = TypeVar("K"), TypeVar("V")
dict[K, V] = {}
		\end{lstlisting}
		
		\pause
		
	\item Es existieren Typen für \tb{besondere} Eingenschaften, wie \pp{Callable}, \pp{Literal}, \pp{Iterable}.
	
	\item[]\begin{lstlisting}[language=pytiny]
choicesT = Literal["yes dear", "no"]  # narrower specification what a string should contain
fn: Callable[[choicesT, int], list | None]  = lambda c, i: [c, i] if i % 2 == 0 else None

fn("yes dear", 2)  # >> ["yes dear", 2]
fn("yes", 1)  # >> None
	\end{lstlisting}
		
	\end{itemize}
	
\end{frame}


\begin{frame}[fragile]{Typing von self}
	\begin{itemize}
		\item Das Typehinten von \pp{self} ist nicht einfach möglich.
		\pause
		\item \pp{self} ist erst \tb{definiert}, wenn die Klasse definiert ist.
		\pause
		\item Während die Klasse also geparsed wird und die Typehints evaluiert werden, ist \pp{self} noch \tb{nicht} bekannt.
		\pause
		\item Abhilfe schafft der \tb{Hilfstyp} \pp{Self} aus dem \pp{typing}-Modul\furl{https://peps.python.org/pep-0673/}.
		
	\end{itemize}
	
\end{frame}


\begin{frame}[fragile]{Anmerkungen}
	\begin{itemize}
		\item Die \pp{Ellipsis} oder \pp{...} ist ein Typ, welcher primär als \tb{Platzhalter} genutzt wird.
		\pause
		
		\item Im Typehinting kommt sie zum Einsatz um
		\pause
		
		\begin{itemize}
			\item ein \pp{tuple}, oder andere Sequenzen \tb{beliebiger} Länge zu beschreiben
			\item[]\begin{lstlisting}[language=pytiny]
my_tuple: tuple[int, ...] = (1, 2, 3)  # (1,) would be also okay
			\end{lstlisting}
			
			\pause
			\item beliebige Parameter-Signaturen einer Funktion zu symbolisieren
			\item[]\begin{lstlisting}[language=pytiny]
# parameters are arbitrary, important is that we get a list
fn: Callable[..., list]  = lambda x, y, z, a, b, c: []  
			\end{lstlisting}

		\end{itemize}
		\pause

\begin{frame}{Function Decorators}
	\begin{itemize}
		\item Um eine Funktion um zusätzliche Funktionalitäten zu ergänzen, verwendet man \tb{Function Decorators}\footnote{Sinnvolle, verständliche Beispiele für Funktion Decorators: \url{https://realpython.com/primer-on-python-decorators/}}.
		\pause
		
		\item Ein Decorator nimmt die zu wrappende Funktion entgegen, fügt um ihrem Aufruf \tb{weitere Funktionalität} hinzu und gibt eine \tb{neue Funktion} zurück.
		\pause
		\item Er wird durch ein \pp{@} \tb{gekennzeichnet} und steht oberhalb der gewrappten Funktion.
		\pause
		\item Er ist ein beliebtes Tool von bekannten \tb{Libraries}, wie \lpl{dataclass}, \lpl{FastAPI} oder auch \pp{discord.py}. 
		\pause
		\item Ein sehr simpler, aber mächtiger Decorator, ist der \pp{@cache}-Decorator\furl{https://docs.python.org/3/library/functools.html\#functools.cache}.
	\end{itemize}
\end{frame}

	\begin{lstlisting}[language=pytiny]
def call_me_twice(old_function: Callable) -> Callable:
    def new_function():
        old_function()
        old_function()

    return new_function

@call_me_twice
def say_cyber():
    print("Cyber!")

say_cyber()  # >> Cyber![NEWLINE]Cyber!
	\end{lstlisting}
	
	Einfach gesagt übergibt man einer \tb{Decoratordefinition} ein Funktionsobjekt
	und nutzt dieses, um ein neues Funktionsobjekt zu erzeugen und zurückzugeben.
	
\end{frame}

\begin{frame}{Warum Mutable Defaults bei Parametern mit Defaultwerten ein Problem sind}
	\begin{itemize}
		\item Die \tb{Standardwerte} für die Parameter werden einmal zu \tb{Beginn} der Laufzeit evaluiert und sind dann \tb{fest}.
		\pause
		\item[] Jedes erstellte \tb{Objekt} einer Klasse erhält also die \tb{selbe} Referenz auf das selbe Objekt.
		\pause
		\item Dies ist kein Problem, wenn das Argument \tb{immutable} ist.
		\pause
		\item Wird eine \pp{list} als Default benötigt, lässt sich diese z.B. als \pp{tuple} realisieren.
		\pause
		\item Ansonsten ist die \tb{gängige} Methode, das Argument auf \pp{None} zu setzen und in der \pp{__init__} dann korrekt zu initialiseren.
	\end{itemize}
\end{frame}

\section{FastAPI}

\subsection{Eine RestAPI - Überblick}

\begin{frame}{REST-Bedingungen\footnote{Folie der 4. POOSE-Vorlesung von Felix Boes entnommen.}}
    \vspace{-0.3ex}
    Eine Webanwendung ist \tb{RESTful}, wenn sie folgende Bedingungen erfüllt. \pause
    \begin{itemize}
      \item Client-Server-Architektur: Der \tb{Dienst} besteht aus Server- und Clientanteilen. \pause
      \item \tb{Zustandslosigkeit}: Anfragen und Antworten sind durch einzelne REST-Nachrichten \tb{vollständig} beschrieben. \pause
        Zwischen Client und Server besteht keine Sitzung (auf der \tb{HTTP}-Kommunikationsebene, aber vlt. auf anderen Kommunikationsebenen). \pause
      \item Caching: Wegen der Zustandslosigkeit liegt es (sehr oft) nahe, Antworten zu cachen. \pause
      \item \tb{Einheitliche} Schnittstelle (fundamentale Designbedingung):
        \begin{itemize}
          \item \tb{Ressourcenadressierung} durch URIs \pause
          \item Ressourcenänderung durch geeignete Ressourcendarstellungen \pause
          \item \tb{Selbsterklärende} Nachrichten \pause
          \item Hypermedia as the Engine of Application State (\tb{HATEOAS}) \pause
        \end{itemize}
      \item Mehrschichtige Systeme: Hinter der Schnittstelle wird der Dienst durch das Zusammenspiel von aufeinander aufbauenden Anwendungen und Diensten realisiert.
    \end{itemize}
  \end{frame}


\begin{frame}[fragile]{Antworten\footnote{Ebenfalls aus der POOSE übernommen.}}
    Aufbau einer \tb{Antwort} (Response):
    
    \begin{lstlisting}[language=pltiny]
  PROTOCOL_VERSION | STATUS_CODE | STATUS_MESSAGE
  CONTENT (Optional)
    \end{lstlisting}\pause
    
    Die zulässigen Statuscodes und Statusnachrichten sind in fünf Gruppen eingeteilt\footnote{Eine vollständige Übersicht findet sich u.A. hier: \url{https://http.cat/}}.
    \begin{itemize}
      \item Informational responses (100-199)
      \item \tb{Successful} responses (200-299)\footnote{FunFact: Der inoffizielle Statuscode 218 \glqq This is fine\grqq\ wird verwendet, um Fehler an einem reverse-Proxy vorbei zu schmuggeln.} Beispiel: \lpl{200 = OK}
      \item Redirection messages (300-399)
      \item \tb{Client} error responses (400-499) Beispiel: \lpl{404 = RESSOURCE NOT FOUND}
      \item \tb{Server} error responses (500-599)
    \end{itemize}\pause
    
    Der Typ des erhaltenen \lpl{CONTENT} ist in der Regel der gewünschte Rückgabetyp.
    
    \note{\LARGE XX:51}
  \end{frame}

\begin{frame}{Parameter}
	\begin{itemize}
		\item \tb{Path-Parameter} können entgegengenommen werden, indem die Signatur der Funktion einen Parameter des \tb{gleichen} Namens enthält.
		\pause
		\item \tb{Query-Parameter} werden normal in der \tb{Funktionssignatur} deklariert.
		\item Sind die Parameter typ-annotiert, so übernimmt FastAPI automatisch die \tb{Validierung} und Konvertierung.
		\pause
		\item Eine Funktion kann auch nur einen \tb{Parameter} \pp{request: fastapi.Request}\furl{https://fastapi.tiangolo.com/advanced/using-request-directly/} entgegennehmen.
		\pause
		\item Dieser enthält den \tb{gesamten} Kontext \pp{path_params}, \pp{query_params}, \pp{cookies}, etc.
		\pause
		\item Durch die Nutzung der \pp{Request}-Klasse, existieren \tb{keine} automatischen Typvalidierungen mehr.
	\end{itemize}
\end{frame}


\begin{frame}{Routen definieren}
	\begin{itemize}
		\item Eine \tb{Route} kann durch den \pp{app.get()}- bzw \pp{app.post()}-\tb{Decorator} definiert werden.
		\pause
		\item Eine weitere Option ist \pp{app.add_api_route(path, endpoint, methods)}.
		\item[] Diese wird insbesondere dann relevant, wenn der Code in \tb{mehrere Dateien} aufgeteilt wird oder wenn \tb{Methoden} als Endpoint genutzt werden sollen.
		\pause
		\item Es ist good practice, alle Routen einer API-Version in einen \tb{Namespace} zu packen.
		\item[] Beispiel: \pp{url:port/api/v1/here_the/route_name}.
	\end{itemize}
\end{frame}


\begin{frame}{Responses}
	\begin{itemize}
		\item Jede Endpoint-Funktion muss eine \tb{Response} zurückgeben.
		\pause
		\item Eine Response\footnote{\url{
        https://fastapi.tiangolo.com/tutorial/response-model/}} kann sein:
		\item[] \begin{itemize}
			\pause
            \item ein \pp{dict}.
            \pause
            \item ein Objekt, welches von \tb{Pydantic}'s \pp{BaseModel} erbt.
            \pause
            \item eine Response aus \pp{fastapi.responses}, z.B. die \pp{JSONResponse}.
            \pause
            \item eine geworfene \pp{fastapi.HTTPException}. %Adrian told me so
            \pause
        \end{itemize}
		\item Die Nutzung des \pp{BaseModel} sorgt für eine bessere \tb{automatisch} erstellte Dokumentation.
		\pause
		% TODO: hierzu ein beipsiel mit Status code
		\item Der Response Typ, sowie der erwartete \tb{Statuscode} können innerhalb der Funktion normal erstellt und zurückgegeben werden.
		\pause
        \item Alternativ können die erwarteten Typen bereits bei der \tb{Registrierung} der Route gesetzt werden.
	\end{itemize}
\end{frame}

import uuid

from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
from pydantic import BaseModel

app = FastAPI()

class User(BaseModel):
    name: str
    id: uuid.UUID | None = None

    def model_post_init(self, ctx):
        if self.id is None:
            self.id = uuid.uuid4()

@app.post("/api/v1/register/{u_name}")
async def read_item(request: Request):
    u = User(name=request.path_params["u_name"])
    return JSONResponse(content=u.model_dump_json(), status_code=200)
    \end{lstlisting}

\subsection{atexit}
\begin{frame}[fragile]{atexit}
	\begin{itemize}
		\item Wird ein Programm (gracefully) \tb{beendet}, bekommt es Zeit, um kontrolliert zu terminieren.
		\pause
		\item Diese \tb{Zeit} kann genutzt werden, um Daten aus dem Speicher auf ein persistentes Medium zu übertragen, erstellte temporäre Dateien zu entfernen, und mehr.
		\pause
		\item Python bietet das \pp{atexit}-Modul für genau diesen Zweck.
		\pause
		\item Dieses bietet einen \tb{Decorator}, sowie eine explizite \pp{register}-Funktion.
		\pause
		\item Hier \tb{registrierte} Funktionen werden beim Signal zu enden ausgeführt.
		\pause
	\end{itemize}
 \begin{lstlisting}[language=pytiny]
@atexit.register
def goodbye():
    print('You are now leaving the Python sector.')	

def goodbye_with_args(name: str, adjective: str):
   print(f"Goodbye {name}, was {adjective} to meet you!")

atexit.register(goodbye_with_args, 'Donny', 'nice')
	\end{lstlisting}
\end{frame}
