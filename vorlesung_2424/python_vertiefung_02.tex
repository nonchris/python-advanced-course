\subsection{Imports von Modulen}

\begin{frame}[fragile]{Pakete und Module einbinden}
        Das gesamte Python-\tb{Ökosystem} baut auf dem Importieren (fremder) Pakete auf.

		\medskip
		
		In Python gibt es \tb{zwei Möglichkeiten}, Pakete einzubinden.
		
		\medskip
		
		\begin{lstlisting}[language=pytiny]
import random
from random import randint
		\end{lstlisting}
		
		\medskip
		
		Erstere Struktur importiert das \tb{gesamte Modul} und stellt \tb{seinen Namespace} objekt-ähnlich zur Verfügung.\footnote{Verändert der eigene Code den Wert einer globalen Variable des Moduls, so ändert er sich überall, wo das Modul mit der \texttt{import}-Struktur importiert ist.}
		
		\medskip
		
		Die \pp{from}-Struktur bindet eine \tb{Referenz} des importierten Objekts an den \tb{gegebenen Namen} innerhalb des importierenden Namespaces.
		
		\medskip
		
		Das Einbinden von fremder Funktionalität via \lpl{from module import *} \pause widerspricht dem Konzept der Namespaceunterteilung und soll nicht verwendet werden.

	
\end{frame}


\begin{frame}{Ein Modul}
        Jede Python-Datei ist ein eigenständiges \pp{Modul}\furl{https://packaging.python.org/en/latest/glossary/\#term-Pure-Module} mit eigenem \tb{Namespace}.

		\medskip
		
		Mehrere \tb{Dateien} in einem Ordner können zu einem \tb{gemeinsamen} \pp{Package}\footnote{\texttt{Packages} werden oft auch in Pythons eigenen Docs als \texttt{Module} bezeichnet. Package ist im Grunde eine spezifischere Bezeichnung für ein Modul, welche etwas über die Struktur verrät. Zudem gilt immer: Ein Package besteht aus ein- bis n-Modulen.} zusammengefasst werden, welches dann \tb{als eins importiert} werden kann.\newline
		%
		Hierzu muss in dem Verzeichnis eine Datei \pp{__init__.py} \footnote{Einer der ganz wenigen Orte, wo \texttt{from x import *} Import sinnvoll sein kann, um alle Elemente anderer Package-Dateien aus init heraus zur Verfügung zu stellen.} liegen.
		
		\medskip
		
		Jedes Modul hat einen \tb{Namen}. Dieser besteht aus \pp{packagename.modulname}.
		
		\medskip
		
		Einzige \tb{Ausnahme} ist die initial ausgeführte Datei, welche einen \tb{zweiten Namen} hat. Dieser ist immer \pp{__main__}\footnote{Das Main-Modul eines durch \texttt{multiprocessing} erzeugten Prozesses heißt \texttt{\_\_mp\_main\_\_}}.

		\nsl{Erst moulname.datei und dann \_\_main nach init bei ausführung.\\ quasi erst import/ init, dann exec}
\end{frame}


\begin{frame}[fragile]{Ausführung und Import eines Moduls}
        Der \tb{Import} eines Moduls sorgt für dessen \tb{komplette Ausführung}.\newline
		%
		Hierbei sollten lediglich \tb{Funktionen und Konstanten} definiert werden.
		
		\medskip
		
		Wird ein Modul hingegen explizit als \tb{Programm aufgerufen}, so soll dieses nicht nur Funktionalitäten definieren, sondern auch eine \tb{Funktionalität ausführen}.
		
		\medskip
		
		Daher soll ein \tb{Modul} in der Lage sein zu \tb{unterscheiden}, ob es der initial ausgeführte Entrypoint, oder ein Import ist.
		
		\medskip
		
		Dies geht über die \tb{Überprüfung} der in \pp{__name__} verfügbaren Information.
		
		\medskip

    \begin{lstlisting}[language=pytiny]
# some imports and functions

if __name__ == '__main__':
    some_functionality()
            \end{lstlisting}
\end{frame}

\begin{frame}{Packages im Projektverzeichnis}
	Die von Python für den möglichen \tb{Import gefundenen} Pakete hängen in Teilen von dem \tb{Arbeitsverzeichnis} ab\footnote{Pakete und Module im Arbeitsverzeichnis werden zu sys.path hinzugefügt, Module in Subverzeichnissen, wie src/, jedoch nicht.}\furl{https://docs.python.org/3/library/sys_path_init.html}.
	
	\medskip
	
	Dies ist insbesondere für den \tb{Import lokaler Pakete} ein Problem.\newline
	\pause
	So können u.A. IDEs Imports unter Umständen nicht auflösen, wenn sich das Paket nicht im Arbeitsverzeichnis, standardmäßig Project-Root, befindet.
	
	\medskip
	\pause
	
	Um ein \tb{einheitliches Verhalten} von \tb{projektinternen Imports} bei
	\pause
	Import des Packages, 
	\pause 
	Ausführung als Terminal-Programm 
	\pause
	und direkter Ausführung der \pp{__init__.py}
	\pause
	zu gewährleisten, sollten sich alle \tb{entwickelten} Pakete direkt im \tb{Project-Root} befinden.
\end{frame}



\begin{frame}[fragile]{Ein Fix bei hirstorisch gewachsenen Projekten mit src/ Verzeichnis}
	
	Eine simple, aber effektive \tb{Lösung} ist die Verwendung dieses Codeblocks, \tb{bevor} die eigenen Imports in der initial-ausgeführten Datei des Packages angegeben werden.
	\pause
	
	
	\begin{lstlisting}[language=pytiny]
# make sure we can import other modules from our package when it is run directly
# package is None when the module is executed directly (and not imported)
# a module is frozen when it's packaged in an executable that doesn't rely on external dependencies
if __package__ is None and not hasattr(sys, "frozen"):
    import os.path

    # get path of the current file
    path = os.path.realpath(os.path.abspath(__file__))
    # add current directory to list of directories interpreter searches when a module is required
    sys.path.insert(0, os.path.dirname(os.path.dirname(path)))

# import things from your package here!
from packaging_sample_project import some_functions
	\end{lstlisting}
	
\end{frame}

\begin{frame}[fragile]{Dependencies spezifizieren I}

	In der \lpl{.toml} gibt es ein Feld namens \lpl{dependencies}, dort können Dependencies angegeben werden. Dieses ist mit der \pp{requirements.txt} kompatibel.
	
	\pause
	\medskip
	
	Dependencies sollten für \tb{Reproduzierbarkeit} immer mit einer Versionsnummer\footnote{wie Packages ihre Versionsnummern angeben kann variieren} angegeben werden.
	\begin{itemize}
		\item Versionsnummern können mit \lpl{==}, \lpl{>=},  \lpl{\~=} spezifiziert werden.
		\item Hierbei bedeutet \lpl{\~=} \glqq kompatibel mit\grqq
	\end{itemize}
	Selbige Art von \tb{Versionpinning} kann auch in der \lpl{requirements.txt} vorgenommen werden\furl{https://pip.pypa.io/en/latest/reference/requirements-file-format/}:
	
	\begin{lstlisting}[language=pytiny]
fastapi==0.103.1
openai-whisper>=20230124¿\footnote{Dieses Listing ist ein Beispiel für die requirements.txt-Syntax. Die .toml Syntax ist leicht anders: \url{https://peps.python.org/pep-0631/\#example}}
	\end{lstlisting}
%
\end{frame}


\begin{frame}[fragile]{Dependencies spezifizieren II}
	Dependencies können \tb{dynamisch} aus der \pp{requirements.txt} in die \pp{.toml} eingelesen werden.
	
	\medskip
	
	Ebenso kann die Version des Projektes aus der \pp{__init__.py} eingelesen werden, um eine \textsl{Single Source of Truth} für die Versionsnummer im Projekt zu haben.
	
	
	\begin{lstlisting}[language=pytiny]
[project]
dynamic = ["dependencies", "version"]

[tool.setuptools.dynamic]
dependencies = { file = ["requirements.txt"] }
version = { attr = "whisper_api.__version__" }
	\end{lstlisting}

\end{frame}


\begin{frame}[fragile]{Entrypoint setzen}

	Wenn das Package \tb{direkt} ausführbaren Code enthält, muss hierfür ein \tb{Entrypoint} in der \lpl{.toml} spezifiziert werden.
		
		\medskip
		
	Dies passiert, indem eine \tb{Funktion} innerhalb des Packages angegeben wird, welche durch einen \tb{Command} aufgerufen werden soll.

	
	
	\begin{lstlisting}[language=pytiny]
[project.scripts]
run_package = "package_name:main"
	\end{lstlisting}
	
	

 Nach dem \tb{Install} via \lpl{pip} kann durch die Eingabe des Commands \lpl{run_package} die main-Funktion des Packages \lpl{package_name} \tb{ausgeführt} werden.

	
\end{frame}

\begin{frame}[fragile]{Eigenes Package installieren und ausführbar machen}
	Ein eigenes Paket kann in eine Python Umgebung \tb{installiert} werden durch:
		
	\medskip
	
	\begin{lstlisting}[language=pytiny]
python3 -m pip install -e .
	\end{lstlisting}
	
	\medskip
	
	\begin{itemize}
		\pause
		\item Unter der Annahme, dass man sich im \tb{Wurzelverzeichnis} des Projektes befindet (dort wo die \lpl{.toml} liegt).
		\pause
		\item \lpl{-e} sorgt dafür, dass das Paket als \tb{editierbar} installiert wird.
		\item[] Dadurch kann der Code einfach weiter entwickelt werden, ohne das Paket dauerhaft neu installieren zu müssen.
		\pause
		\item[] Diese Flag sollte \tb{nicht} im Deployment verwendet werden!
	\end{itemize}
	
	\medskip
	
	Wie bei allen \tb{Installationen} von Python-Paketen sollte auch hier darauf geachtet werden, in einem \tb{venv}\furl{https://docs.python.org/3/library/venv} zu arbeiten!

	
\end{frame}

\subsection{Weitere wichtige Hinweise zum packaging}

\begin{frame}{\_\_init\_\_.py}
	Ist die \pp{__init__.py} \tb{nicht leer}, sind nach außen nur \tb{Funktionalitäten sichtbar}, welche in dieser deklariert oder \tb{importiert} werden.
	
	\medskip
	\pause
	
	Um unerwartete \tb{Seiteneffekte} zu \tb{vermeiden}, sollte sich die \pp{__init__.py} auf folgendes \tb{beschränken}:	
	\medskip
	
	
	\pause
	\tb{Mindestens}:
	\pause
	\begin{itemize}
		\item Deklaration von Metadaten
		\pause
		\item Import von Funktionalität, die bei Import zur Verfügung gestellt werden soll.
	\end{itemize}
	\pause
	\tb{Optional}:
	\pause
	\begin{itemize}
		\item Definieren einer Entrypoint-Funktion für die Ausführung des Projektes
		\pause
		\item Eine \pp{__main__}-Clause
	\end{itemize}
	\nsl{Nicht jede \_\_init\_\_.py muss oder sollte all diese Aspekte enthalten.}
\end{frame}


\begin{frame}{setup.py}
	\begin{itemize}
		\item Ist eine \tb{dynamische} Art und Weise, die Daten aus der \lpl{.toml} anzugeben, die in vielen Projekten zu finden ist.

		\item[] Motivation: Vereinfachung und \tb{Vereinheitlichung} der Spezifikation von\\ Projekt-Metadaten.
		\item Beide Wege, ein Projekt zu spezifizieren fühlen sich sehr \tb{ähnlich} an. Ist ein Weg klar, so ist der andere auch verständlich!
	\end{itemize}	
\end{frame}

\subsection{Abstrakt}
\begin{frame}{Erinnerung: Warum Typehinting}
	\begin{itemize}
		% TODO Spiderman meme von variable auf str auf int
		\item Python ist \tb{dynamisch} typisiert. Eine Variable kann alles sein.
		\item Für mehr \tb{Nachvollziehbarkeit} des Codes wurde Typehinting eingeführt.\footnote{PEP484: \url{https://peps.python.org/pep-0484/}, eingeführt mit python3.5}
		\item Typehints geben den \tb{erwarteten} Typ bzw. die \tb{möglichen Typen} einer Variable an.
		\item[] Dieses hilft
		\begin{itemize}
			\item beim \tb{Dokumentieren} des eigenen Codes für sich selbst und für andere.
			\item der \tb{IDE} dabei, den Code zu verstehen und sinnvollere \tb{Vorschläge} zu machen.
			\item Libraries dabei, \tb{automatische} Checks und Conversions vorzunehmen.
			\item statischen \tb{Typecheckern}, wie \pp{mypy}\furl{https://mypy-lang.org/}, dabei, unseren Code zu testen.
		\end{itemize}
	\end{itemize}	
\end{frame}


\begin{frame}{Was machen Typehints intern?}

		
		% TODO: soll dieser Punkt rein und ist der korrekt so? - nicht hier. und gilt nur partiell fir zb Iterable

\end{frame}

\begin{frame}[fragile]{Typehinting - Grundlagen \#2}	
        Sollen \tb{mehrere} Typen erlaubt sein, erlaubt python die Notation:
		\footnote{Diese Notation ist ab python3.10 verfügbar.  In früheren Versionen werden die Hilfsklassen \texttt{Union} und ggf. \texttt{Optional} aus dem \texttt{typing} Modul benötigt.}

		\end{lstlisting}%\pause
		
		\medskip

def my_path_fn(arg1: str | Path) -> Path | None:

		\end{lstlisting}
		
		\medskip
		
		Alle bekannten built-in \tb{Datentypen} können als Typehint verwendet werden.\newline
		%
		Dabei können \glqq generische\grqq~ Typen noch \tb{genauer} spezifiziert werden.
		\footnote{Diese Notation ist ab python3.9 verfügbar. In früheren Versionen werden die Hilfsklassen \texttt{List}, \texttt{Dict}, etc aus \texttt{typing} benötigt.}
		
		\begin{lstlisting}[language=pytiny]

\subsection{Eine RestAPI - Überblick}

\begin{frame}{REST-Bedingungen\footnote{Folie der 4. POOSE-Vorlesung von Felix Boes entnommen.}}
    \vspace{-0.3ex}
    Eine Webanwendung ist \tb{RESTful}, wenn sie folgende Bedingungen erfüllt. \pause
    \begin{itemize}
      \item Client-Server-Architektur: Der \tb{Dienst} besteht aus Server- und Clientanteilen. \pause
      \item \tb{Zustandslosigkeit}: Anfragen und Antworten sind durch einzelne REST-Nachrichten \tb{vollständig} beschrieben. \pause
        Zwischen Client und Server besteht keine Sitzung (auf der \tb{HTTP}-Kommunikationsebene, aber vlt. auf anderen Kommunikationsebenen). \pause
      \item Caching: Wegen der Zustandslosigkeit liegt es (sehr oft) nahe, Antworten zu cachen. \pause
      \item \tb{Einheitliche} Schnittstelle (fundamentale Designbedingung):
        \begin{itemize}
          \item \tb{Ressourcenadressierung} durch URIs \pause
          \item Ressourcenänderung durch geeignete Ressourcendarstellungen \pause
          \item \tb{Selbsterklärende} Nachrichten \pause
          \item Hypermedia as the Engine of Application State (\tb{HATEOAS}) \pause
        \end{itemize}
      \item Mehrschichtige Systeme: Hinter der Schnittstelle wird der Dienst durch das Zusammenspiel von aufeinander aufbauenden Anwendungen und Diensten realisiert.
    \end{itemize}
  \end{frame}


\begin{frame}[fragile]{Antworten\footnote{Ebenfalls aus der POOSE übernommen.}}
    Aufbau einer \tb{Antwort} (Response):
    
    \begin{lstlisting}[language=pltiny]
  PROTOCOL_VERSION | STATUS_CODE | STATUS_MESSAGE
  CONTENT (Optional)
    \end{lstlisting}\pause
    
    Die zulässigen Statuscodes und Statusnachrichten sind in fünf Gruppen eingeteilt\footnote{Eine vollständige Übersicht findet sich u.A. hier: \url{https://http.cat/}}.
    \begin{itemize}
      \item Informational responses (100-199)
      \item \tb{Successful} responses (200-299)\footnote{FunFact: Der inoffizielle Statuscode 218 \glqq This is fine\grqq\ wird verwendet, um Fehler an einem reverse-Proxy vorbei zu schmuggeln.} Beispiel: \lpl{200 = OK}
      \item Redirection messages (300-399)
      \item \tb{Client} error responses (400-499) Beispiel: \lpl{404 = RESSOURCE NOT FOUND}
      \item \tb{Server} error responses (500-599)
    \end{itemize}\pause
    
    Der Typ des erhaltenen \lpl{CONTENT} ist in der Regel der gewünschte Rückgabetyp.

  \end{frame}

\begin{frame}{FastAPI - Übersicht}
	    FastAPI bietet alle \tb{Tools}, um einen eine besagte RestAPI zu bauen.
	
		\medskip
		
		Dies geschieht hauptsächlich durch das \tb{Definieren} von Routen.
		
		\medskip
		
		Für jede Route wird eine \tb{Funktion} (\pp{endpoint}) registriert, welche bei \tb{Anfrage} an die Route ausgeführt wird.
		
		\medskip
		
		Durch die Nutzung von \pp{Startlette}\furl{https://www.starlette.io/} und \pp{Pydantic} ist FastAPI sehr \tb{perfomant}.
		
		\medskip
		
		Ausgeführt wird die Anwendung via \pp{uvicorn}\furl{https://www.uvicorn.org/}.
		
		\medskip
		
		Eine automatische \tb{Dokumentation} der API steht unter \pp{[url]/docs} zur Verfügung.

from fastapi import FastAPI

app = FastAPI()  # the main webserver

async def read_root():

def read_item(item_id: int, q: str | None = None):
    return {"item_id": item_id, "q": q}

\begin{frame}{Parameter}
	    \tb{Path-Parameter} können entgegengenommen werden, indem die Signatur der Funktion einen Parameter des \tb{gleichen} Namens enthält.
	
		\medskip
		
		\tb{Query-Parameter} werden normal in der \tb{Funktionssignatur} deklariert.\newline
		%
		Sind die Parameter typ-annotiert, so übernimmt FastAPI automatisch die \tb{Validierung} und Konvertierung.
		
		\medskip
		
		Eine Funktion kann auch nur \tb{einen Parameter} \pp{request: fastapi.Request}\furl{https://fastapi.tiangolo.com/advanced/using-request-directly/} entgegennehmen.\newline
		%
		Dieser enthält den \tb{gesamten} Kontext \pp{path_params}, \pp{query_params}, \pp{cookies}, etc.\newline
		%
		Durch die Nutzung der \pp{Request}-Klasse, existieren \tb{keine} automatischen Typvalidierungen mehr.

\end{frame}


\begin{frame}{Routen definieren}
        Eine \tb{Route} kann durch den \pp{app.get()}- bzw \pp{app.post()}-\tb{Decorator} definiert werden.
        
        \medskip
        
        Eine weitere Option ist \pp{app.add_api_route(path, endpoint, methods)}.\newline
		%
        Diese wird insbesondere dann relevant, wenn die \tb{Routen} in \tb{mehrere Dateien} aufgeteilt werden oder wenn \tb{Methoden} als Endpoint genutzt werden sollen.
        
        \medskip
        
        Für das Aufteilen von Endpoints bietet FastAPI die \tb{APIRouter}\furl{https://fastapi.tiangolo.com/reference/apirouter/}-Klasse.\footnote{Ein Reallife Beispiel: \url{https://github.com/MayNiklas/whisper_api/blob/pydantic-v2-migration/src/whisper_api/api_endpoints/endpoints.py}}
        
        \medskip
        
        Es ist good practice, alle Routen einer API-Version in einen \tb{Namespace} zu packen.\newline
        Beispiel: \pp{url:port/api/v1/here_the/route_name}.
        
        \medspace
        

        
        
        

\end{frame}


\begin{frame}{Responses}
        Jede Endpoint-Funktion muss eine \tb{Response} zurückgeben.

		\medskip
		
		Eine Response\footnote{\url{https://fastapi.tiangolo.com/tutorial/response-model/}} kann sein:
		
		\medskip
		
		\begin{itemize}
			\pause
			\item ein \pp{dict}.
			\pause
			\item ein Objekt, welches von \tb{Pydantic}'s \pp{BaseModel} erbt.
			\pause
			\item eine Response aus \pp{fastapi.responses}, z.B. die \pp{JSONResponse}.
			\pause
			\item eine geworfene \pp{fastapi.HTTPException}. %Adrian told me so
			\pause
		\end{itemize}
		
		\medskip
		
		\pp{BaseModel} sorgt für eine bessere \tb{automatisch} erstellte Dokumentation\footnote{Das trifft auch auf Objekte zu, welche von BaseModel erben.}.
		
		\medskip
		
		% TODO: hierzu ein beipsiel mit Status code
		
		Der \tb{Response Typ}, sowie der erwartete \tb{Statuscode} können innerhalb der Funktion normal \tb{erstellt} und zurückgegeben werden.
		
		\medskip
		
		Alternativ können die erwarteten Typen bereits bei der \tb{Registrierung} der Route gesetzt werden.

\end{frame}

\begin{frame}[fragile]{FastAPI - Zweites Beispiel}
    \begin{lstlisting}[language=pytiny]
import uuid
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
from pydantic import BaseModel

app = FastAPI()

class User(BaseModel):
    name: str
    id: uuid.UUID | None = None

    def model_post_init(self, ctx):
        if self.id is None:
            self.id = uuid.uuid4()

@app.post("/api/v1/register/{u_name}")
async def read_item(request: Request):
    u = User(name=request.path_params["u_name"])
    # could also be just 'return u' in case of a 200 response
    return JSONResponse(content=u.model_dump_json(), status_code=200)
    \end{lstlisting}
\end{frame}

async def lifespan(app: FastAPI):

    start_some_threads()
    yield
    # kill the things, api has shut down
    join_the_threads()
app = FastAPI(lifespan=lifespan)  # register lifespan function
	\end{lstlisting}

\subsection{atexit}
\begin{frame}[fragile]{atexit}
        Wird ein Programm (gracefully) \tb{beendet}, bekommt es Zeit, um kontrolliert zu terminieren.

		\medskip
		
		Diese \tb{Zeit} kann genutzt werden, um Daten aus dem Speicher auf ein persistentes Medium zu übertragen, erstellte temporäre Dateien zu entfernen, und mehr.
		
		\medskip
		
		Python bietet das \pp{atexit}-Modul für genau diesen Zweck.
		
		\medskip
		
		Dieses bietet einen \tb{Decorator}, sowie eine explizite \pp{register}-Funktion.
		
		\medskip
		
		Hier \tb{registrierte} Funktionen werden beim Signal zu enden ausgeführt.
		
		\medskip

 \begin{lstlisting}[language=pytiny]
@atexit.register
def goodbye():
    print('You are now leaving the Python sector.')	

def goodbye_with_args(name: str, adjective: str):
   print(f"Goodbye {name}, was {adjective} to meet you!")

atexit.register(goodbye_with_args, 'Donny', 'nice')
	\end{lstlisting}
\end{frame}
