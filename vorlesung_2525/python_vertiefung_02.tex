\subsection{Imports von Modulen}

\begin{frame}[fragile]{Importing Packages and Modules}
        The entire Python \tb{ecosystem} is based on importing (external) packages.

		\medskip
		
		In Python there are \tb{two ways} to import packages.
		
		\medskip
		
		\begin{lstlisting}[language=pytiny]
import random
from random import randint
		\end{lstlisting}
		
		\medskip
		
		The first structure imports the \tb{entire module} and makes \tb{its namespace} available in an object-like way.\footnote{If your own code changes the value of a global variable in the module, this change is visible everywhere the module was imported with the \texttt{import} structure.}
		
		\medskip
		
		The \pp{from}-structure binds a \tb{reference} of the imported object to the \tb{given name} within the importing namespace.
		
		\medskip
		
		Importing external functionality via \lpl{from module import *} \pause contradicts the concept of namespace separation and should not be used.

\end{frame}


\begin{frame}{A Module}
        Every Python file is an independent \pp{module}\furl{https://packaging.python.org/en/latest/glossary/\#term-Pure-Module} with its own \tb{namespace}.

		\medskip
		
		Multiple \tb{files} in one folder can be grouped into a \tb{single} \pp{package}\footnote{\texttt{Packages} are often also called \texttt{modules} in Python’s own docs. A package is basically a more specific term that says something about structure. In essence: a package consists of one or more modules.} which can then \tb{be imported as one}.\newline
		%
		For this, the directory must contain an \pp{__init__.py} file\footnote{One of the very few places where \texttt{from x import *} can make sense, to expose all elements of other package files from init.}.
		
		\medskip
		
		Every module has a \tb{name}. This consists of \pp{packagename.modulename}.
		
		\medskip
		
		The \tb{initially executed} module has a \tb{second} name: \pp{__main__}\footnote{The main module of a process created by \texttt{multiprocessing} is called \texttt{\_\_mp\_main\_\_}, which is useful to detect if you're a child process or the main one}.
	

		\nsl{First modulename.file, then \_\_main after init when executed.\\ basically first import/init, then exec}
\end{frame}


\begin{frame}[fragile]{Execution and Import of a Module}
        Importing a module triggers its \tb{complete execution}.\newline
		%
		This execution should \tb{only} define \tb{functions and constants}.
		
		\medskip
		
		A module should \tb{only} execute functionality if it is explicitly run as a \tb{program}.
		
		\medskip
		
		Thus a \tb{module} should be able to \tb{distinguish} whether it is the initially executed entry point or an import.
		
		\medskip
		\pause
		
		This works by \tb{checking} the information available in \pp{__name__}.
		
		\medskip

    \begin{lstlisting}[language=pytiny]
# some imports and functions

if __name__ == '__main__':
    some_functionality()
            \end{lstlisting}
\end{frame}

\begin{frame}{Packages in the Project Directory}
	Packages that Python can potentially \tb{import} depend partly on the \tb{working directory}\footnote{Packages and modules in the working directory are added to sys.path, modules in subdirectories like src/ are not.}\furl{https://docs.python.org/3/library/sys_path_init.html}.
	
	\medskip
	
	This is a problem for \tb{importing local packages} in particular.\newline
	\pause
	IDEs, among others, may not resolve imports if the package is not in the working directory, which by default is the project root.
	
	\medskip
	\pause
	
	To ensure \tb{consistent} behavior for \tb{internal and external} project \tb{imports} 
	of the package, 
	\pause 
	running it as a terminal program, 
	\pause
	or executing the \pp{__init__.py} directly, 
	\pause
	all \tb{developed} packages should be placed directly in the \tb{project root}.
\end{frame}



\begin{frame}[fragile]{A Fix for Projects that evolved over time with src/ Directory}
	
	A simple yet effective \tb{solution} is to use this code block \tb{before} specifying your own imports in the initially executed file of the package.
	\pause
	
	
	\begin{lstlisting}[language=pytiny]
# make sure we can import other modules from our package when it is run directly
# package is None when the module is executed directly (and not imported)
# a module is frozen when it's packaged in an executable that doesn't rely on external dependencies
if __package__ is None and not hasattr(sys, "frozen"):
    import os.path

    # get path of the current file
    path = os.path.realpath(os.path.abspath(__file__))
    # add current directory to list of directories interpreter searches when a module is required
    sys.path.insert(0, os.path.dirname(os.path.dirname(path)))

# import things from your package here!
from packaging_sample_project import some_functions
	\end{lstlisting}
	
\end{frame}

\begin{frame}[fragile]{requirements.txt}
	The probably most widely used way to \tb{specify dependencies}.\newline
	\pause
	There is \tb{no real difference} compared to the \pp{.toml}; packages either:
	\begin{itemize}
		\item Specify dependencies directly in the \pp{.toml}
		\pause

		\pause

	\end{itemize}
	
	\medskip
	\pause
	
	With \pp{requirements.txt}, a widely \tb{known standard} for specifying dependencies has emerged, which is much better known and \tb{easier} to use than specifying in the \pp{.toml}.
	
	\medskip

	\medskip

python3 -m pip install -r requirements.txt¿\footnote{\texttt{python -m} to run \texttt{pip} ensures that the always included module \texttt{pip} is executed instead of requiring pip as an executable.}¿
	\end{lstlisting}

\begin{frame}[fragile]{Specifying Dependencies I}

	In the \lpl{.toml} there is a field called \lpl{dependencies}, where dependencies can be specified. This is compatible with the \pp{requirements.txt}.
	
	\pause
	\medskip
	
	Dependencies should always include a version number\footnote{The way packages specify their version numbers may vary} for \tb{reproducibility}.
	\begin{itemize}
		\item Version numbers can be specified with \lpl{==}, \lpl{>=}, \lpl{\~=}.
		\item Here, \lpl{\~=} means "compatible with".
	\end{itemize}
	The same kind of \tb{version pinning} can also be done in \lpl{requirements.txt}\furl{https://pip.pypa.io/en/latest/reference/requirements-file-format/}:
	
	\begin{lstlisting}[language=pytiny]
fastapi==0.103.1
openai-whisper>=20230124¿\footnote{This listing is an example for the requirements.txt syntax. The .toml syntax is slightly different: \url{https://peps.python.org/pep-0631/\#example}}
	\end{lstlisting}
%
\end{frame}



\begin{frame}[fragile]{Specifying Dependencies II}
	Dependencies can be \tb{dynamically} read from the \pp{requirements.txt} into the \pp{.toml}.
	
	\medskip
	
	Similarly, the project version can be read from \pp{__init__.py} to have a \textsl{single source of truth} for the version number in the project.
	
	\begin{lstlisting}[language=pytiny]
[project]
dynamic = ["dependencies", "version"]

[tool.setuptools.dynamic]
dependencies = { file = ["requirements.txt"] }
version = { attr = "whisper_api.__version__" }
	\end{lstlisting}

\pause

You can also declare \tb{optional} dependencies\furl{https://packaging.python.org/en/latest/guides/writing-pyproject-toml/\#dependencies-and-requirements}, e.g. tools that only the library devs need.

\end{frame}



\begin{frame}[fragile]{Setting the Entrypoint}

	If the package contains \tb{directly} executable code, an \tb{entrypoint} must be specified in the \lpl{.toml}.
		
		\medskip
		
	This is done by specifying a \tb{function} within the package that should be called via a \tb{command}.
	
	
	
	\begin{lstlisting}[language=pytiny]
[project.scripts]
run_package = "package_name:main"
	\end{lstlisting}
	
	After installing via \lpl{pip}, entering the command \lpl{run_package} in the terminal executes the main function of the package \lpl{package_name}.
\end{frame}


\begin{frame}[fragile]{Installing your own Package and making it executable}
	You can \tb{install} your own package into a Python environment by:
		
	\medskip
	
	\begin{lstlisting}[language=pytiny]
python3 -m pip install -e .
	\end{lstlisting}
	
	\medskip
	
	\begin{itemize}
		\pause
		\item Assuming you are in the \tb{root directory} of the project (where the \lpl{.toml} is located).
		\pause
		\item \lpl{-e} ensures that the package is installed as \tb{editable}.
		\item[] Allowing the code to be developed further without reinstalling the package each time.
		\pause
		\item[] This flag should \tb{not} be used in deployment!
	\end{itemize}
	
	\medskip
	
	As with all Python package \tb{installations}, it is recommended to work in a \tb{venv}\furl{https://docs.python.org/3/library/venv}.

	
\end{frame}

\subsection{Additional Important Notes on Packaging}

\begin{frame}{\_\_init\_\_.py}
	An \tb{non-empty} \pp{__init__.py} exposes only \tb{functionalities}, that are declared or \tb{imported} in it, if the package itself is imported.
	\nll{Ofc its possible to import sub modules}

	\medskip
	\pause
	
	To avoid unexpected \tb{side effects}, the \pp{__init__.py} should be \tb{limited} to the following:	
	
	\medskip
	%
	\pause
	\tb{At a minimum}:
	\pause
	\begin{itemize}
		\item Declaration of metadata
		\pause
		\item Import of functionality that should be available upon import
	\end{itemize}
	\pause
	\tb{Optional}:
	\pause
	\begin{itemize}
		\item Define an entrypoint function for executing the project
		\pause
		\item A \pp{__main__}-clause
	\end{itemize}
	\nsl{Not every \_\_init\_\_.py has to or should include all these aspects.}
\end{frame}

\begin{frame}{setup.py}
	\begin{itemize}
		\item Is a \tb{dynamic} way to specify the data from the \lpl{.toml}, which can be found in many projects
		\item Since PEP 621\furl{https://peps.python.org/pep-0621/} \tb{deprecated}
		\item[] Motivation: Simplification and \tb{standardization} of project metadata specification
		\item Both ways of specifying a project feel very \tb{similar}. If you understood one way, the other is also understandable!
	\end{itemize}	
\end{frame}

\section{Typehinting II}
\subsection{Abstract}
\begin{frame}{Reminder: Why Typehinting}
	\begin{itemize}
		% TODO Spiderman meme from variable to str to int
		\item Python is \tb{dynamically} typed. A variable can be anything.
		\item For more \tb{traceability} of the code, typehinting was introduced.\footnote{PEP484: \url{https://peps.python.org/pep-0484/}, introduced with python3.5}
		\item Typehints indicate the \tb{expected} type or the \tb{possible types} of a variable.
		\item[] This helps
		\begin{itemize}
			\item when \tb{documenting} your own code, for yourself and for others.
			\item the \tb{IDE} to understand the code and make more meaningful \tb{suggestions}.
			\item libraries to perform \tb{automatic} checks and conversions.
			\item static \tb{typecheckers}, such as \pp{mypy}\furl{https://mypy-lang.org/}, to test our code.
		\end{itemize}
	\end{itemize}	
\end{frame}



\begin{frame}{What do Typehints do internally?}

		% TODO: Should this point be included and is it correct this way? – not here. And applies only partially for e.g. Iterable
		% \item In principle, TypeHints only communicate which special methods and other functionalities an object provides.
		
        Typehints are \tb{not checked at runtime}, but remain available and can be \tb{read out}, e.g. using the \pp{inspect}\furl{https://docs.python.org/3/library/inspect.html\#introspecting-callables-with-the-signature-object} module or \pp{obj.__annotations__}.

		\medskip
		
		Some \tb{libraries} actively make use of this, for example to detect signatures, or create and \tb{validate} data schemas.
		
		\medskip
		
		With the help of typehints, function overloading à la Java can be implemented natively\furl{https://youtu.be/yWzMiaqnpkI?t=483}. However, with significant performance drawbacks.

\end{frame}



\subsection{Concrete}
\begin{frame}[fragile]{Reminder: Typehinting – The Basics}
        Basically, the intended type of a \tb{variable} is specified as follows:

		\medskip
		
		\begin{lstlisting}[language=pytiny]

		\end{lstlisting}%\pause
		
		\medskip
		
		\tb{Function parameters} can also be annotated with typehinting. %\pause
		
		\medskip
		
		Similarly, the intended \tb{return type}:
		
		\medskip
		
		\begin{lstlisting}[language=pytiny]
def f(param: type, param_2: type) -> returntype:
    ...
		\end{lstlisting} %\pause
		
		\medskip
		
		Primary source for this chapter: \url{https://docs.python.org/3/library/typing}
	
\end{frame}

\begin{frame}[fragile]{Advanced Typing}
    \tb{Types} can also be stored in \tb{variables} (\pp{Type aliases}).
		\footnote{There is no naming convention for aliases. A common notation is \texttt{typenameT}. Python’s docs do not distinguish between an alias and a “normal” variable.}
		%
		\begin{lstlisting}[language=pytiny]
numberT = int | float  # this is technically a bad example, because float itself also allows int
x: numberT = 3.14
		\end{lstlisting}
		
		\medskip
		
		\tb{Additional} types can be created with \pp{TypeVar}\footnote{Also from the \texttt{typing} module}.\newline
		%
		This is useful to create, for example, \tb{generic} types \pp{TypeVar("T")}.
		%
		\begin{lstlisting}[language=pytiny]
K, V = TypeVar("K"), TypeVar("V")
dict[K, V] = {}
		\end{lstlisting}
		
		\medskip
		
		There are types for \tb{special} properties, such as \pp{Callable}, \pp{Literal}, \pp{Iterable}.
		%
		\begin{lstlisting}[language=pytiny]
choicesT = Literal["yes dear", "no"]  # narrower specification what a string should contain
fn: Callable[[choicesT, int], list | None]  = lambda c, i: [c, i] if i % 2 == 0 else None
fn("yes dear", 2)  # >> ["yes dear", 2]
fn("yes", 1)  # >> None
		\end{lstlisting}

	
\end{frame}


\begin{frame}[fragile]{Typing self}
	\begin{itemize}
		\item Typehinting \pp{self} is not that easy
		\pause
		\item \pp{self} is only \tb{defined} once the class is defined
		\pause
		\item While the class is being parsed and the typehints are evaluated, \pp{self} is still \tb{unknown}
		\pause
		\item This can be bypassed by declaring a \tb{literal}\furl{https://peps.python.org/pep-0484/\#forward-references}: \pp{-> "MyClass":}
		\pause
		\item As of Python 3.11, the \tb{helper type} \pp{Self} from the \pp{typing} module\furl{https://peps.python.org/pep-0673/} provides a solution
	\end{itemize}
\end{frame}


\begin{frame}[fragile]{Notes}
 	\pp{Ellipsis} or \pp{...} is a type primarily used as a \tb{placeholder}.
	
	\medskip
	
	In typehinting, it is used to
	
	\medskip

		\end{lstlisting}
		
	\end{itemize}

\begin{frame}{Function Decorators}
          To add additional functionality to a function we use 
        \tb{function decorators}\footnote{Meaningful, easy-to-understand examples of function decorators: \url{https://realpython.com/primer-on-python-decorators/}}.

		\medskip
		
		A decorator takes a \tb{function} to be wrapped as an \tb{argument}, defines a \tb{new} function, that \tb{adds functionality} around a call of the original function,
		and returns a \tb{reference} to the \tb{new function}.
		
		\medskip
		
		It is \tb{used} with an \pp{@} and placed above the wrapped function.
		
		\medskip
		
		It is a popular tool in well-known \tb{libraries}, such as \lpl{dataclass}, \lpl{FastAPI}, or \pp{discord.py}.
		
		\medskip
		
		A very simple yet powerful decorator is the \pp{@cache}-decorator\furl{https://docs.python.org/3/library/functools.html\#functools.cache}.
\end{frame}


\begin{frame}[fragile, noframenumbering]{Function Decorators - A simple Example}
	\begin{lstlisting}[language=pytiny]
def call_me_twice(old_function: Callable) -> Callable:
    def new_function():
        old_function()
        old_function()

    return new_function

@call_me_twice
def say_cyber():
    print("Cyber!")

say_cyber()  # >> Cyber![NEWLINE]Cyber!
	\end{lstlisting}
	
	Simply put, you pass a function object to a \tb{decorator definition}
	and use it to create and return a new function object.
	
	\medskip
	
	\tb{Nested} function have \tb{access} to the outer functions \tb{scope}, even if the outer function has already returned.
	
\end{frame}

\begin{frame}{Pydantic - Overview}

\begin{frame}[fragile]{Pydantic - How To}
	Pydantic builds around its class \pp{BaseModel}.
	
	\medskip
	
	\tb{Classes} that should be validated by Pydantic \tb{inherit} from it.\newline
	They declare their \tb{instance} variables \tb{as class} variables\footnote{You can declare as \texttt{ClassVar} (from typing) to preserve their state as class-variable: \url{https://docs.pydantic.dev/latest/concepts/models/\#automatically-excluded-attributes}}.
	
	\medskip
	
	Pydantic classes can be treated as \tb{normal} classes. They can hold arbitrary methods.\newline
	Most modern IDEs should be able to detect Pydantic usage and offer working intelli-sense for instance variables.
	
	\medskip
	
	The auto-generated constructor is \tb{keyword}-argument \tb{only}.

\end{frame}

\begin{frame}{Pydantic Details I - Post-Init Method}

		Pydantic \tb{creates} the \pp{\_\_init\_\_} function \tb{automatically}, making it difficult to attatch functionality to it.
		
		\medskip
		\pause
		
		For this reason Pydantic offers the method\newline \pp{model_post_init(self, \_\_context: )}, which is called after  all attributes are set. This method is meant to be overridden. 
		\pause
		
		\medskip
		
		This provides a \tb{hook} to perform additional work during the init process (e.g., content validation of the variables).
		\pause
		
		\medskip

		If a \pp{super} constructor needs to be called, it also happens here.
\end{frame}


\begin{frame}{Pydantic Details II - Neat Functions}
	
		Selected additional features\footnote{More on BaseModel: \url{https://docs.pydantic.dev/latest/usage/models/\#model-methods-and-properties}}:
	\begin{itemize}
		\item \pp{<BaseModel>.model_dump()} returns a \pp{dict} of the created fields\footnote{The counterpart for serialization is the class method MyModel.validate(dumped\_dict)}.
		\item \pp{<BaseModel>.model_dump_json()} returns a \pp{JSON} string.
		\item A model can hold a field of the type of another class inherriting from \pp{BaseModel}.
		\item A drop-in replacement for Python's \pp{dataclass} decorator is available\footnote{This is not a complete replacement for Pydantic's BaseModel. See: \url{https://github.com/pydantic/pydantic/issues/710}}.
	\end{itemize}
	
\end{frame}


\begin{frame}{Pydantic Details III - The Field}
	The \tb{Field} object allows for a \tb{more precise} specification of fields\furl{https://docs.pydantic.dev/latest/concepts/fields/}:\pause  
	\begin{itemize}
		\item An immutable default value is defined using \pp{default}.
		\item Using \pp{default_factory}, a function can be provided which generates the default for each object instantiation.
		The factory can be \tb{any} function with an empty signature.
		\nll{So it can be arbitrarily complex.\\Methods of the class are not possible bc class is not yet defined.}
		\item With \pp{exclude=True}, fields can be excluded from serialization.
		\item \tb{Aliases} for fields can be defined\furl{https://docs.pydantic.dev/latest/concepts/fields/\#field-aliases}.
	\end{itemize}
\end{frame}


\begin{frame}[fragile]{Pydantic Details IV - Model Validators}
	
	Pydantic allows you to specify additional \tb{validators} that can be executed either \tb{before} or \tb{after} Pydantics own data validation.
	
	\medskip
	
	The before-validation hook can be useful if default values are more complicated
	
	\begin{lstlisting}[language=pytiny]
class GuessingGame(BaseModel):

    possible_words: list[str]

    @model_validator(mode="before")
    @classmethod
    def _fill_target_list_if_none(cls, data) -> None | dict:
         # note: we don't have the object yet. we have a data dict
         # this dict are essentialy all the kwargs that were passed into the constructor
         if data.get("possible_words") is None:
             data["possible_words"] = get_possible_words()

         return data  # we return the 'manipulated' data
	\end{lstlisting}
	
	Note: Pydantic will still require all entries to exist prior to this validator.\nll{Default value or kwarg is still required. \\if not wanted use default factory...}
	
\end{frame}

\begin{frame}{Properties - Motivation}
        Sometimes data is stored in an \tb{internally} useful form, but one that's externally not convenient to use (keyword: timestamp vs. datetime).
		
		
		\medskip
		
		It would be desirable to make this data more \tb{accessible externally},\newline
		%
		without having to keep this representation permanently up-to-date.
		
		\medskip
		
		Similarly, when \tb{setting} an attribute, it may be useful to perform such a \tb{conversion} or checks in the opposite direction\footnote{Without falling into the Java-like getter+setter jungle}.
		
		\medskip
		
		There are also cases where variables should perhaps be made available as \tb{read-only}.
		
		\medskip
		
		All of these problems are solved by the \pp{@property} \tb{decorator}\furl{https://docs.python.org/3/library/functions.html\#property}.
\end{frame}

\subsection{Example}

\begin{frame}[fragile]{A simple Getter / Setter Example}
	
	\begin{lstlisting}[language=pytiny]
class Temperature:
    """Stores a temperature, provides access to valuein celsius and fahrenheit"""

    def __init__(self, t_celsius: float):
        self.celsius = t_celsius

    @property
    def fahrenheit(self):
        return 9*self.celsius/5 + 32

    @fahrenheit.setter
    def fahrenheit(self, t_fahrenheit):
        self.celsius = self.fahrenheit_to_celsius(t_fahrenheit)

    @staticmethod
    def fahrenheit_to_celsius(f):
        return 5*(f-32)/9
	\end{lstlisting}
	
\end{frame}


\begin{frame}{Properties - Notes}
	     Property getter calls should not have \tb{unexpected effects} on the object!
		
		\medskip
		
		When writing the class, one must pay close \tb{attention} to whether the internal or external \tb{version} of an attribute is being accessed.
		
		\medskip
		
		If a property getter \tb{waits} for I/O or a \tb{lock}, then the (seemingly harmless) access to an attribute can cause the program to \glqq freeze\grqq!
		
		\medskip
		
		Property \tb{setters} can \tb{quickly become confusing} if they implement more functionality than simple type checks/conversions!\newline
		%
		If they need to do more than this, it is advisable to instead use a \glqq classic\grqq~(setter) function.\footnote{In general, properties are just nice syntactic sugar. There is nothing wrong with working with getters and setters in the classic way.}
\end{frame}

\subsection{A RestAPI - Overview}

\begin{frame}{REST Conditions\footnote{Slide taken from the 4th POOSE lecture by Felix Boes and translated to English.}}
    \vspace{-0.3ex}
    A web application is \tb{RESTful} if it meets the following conditions. \pause
    \begin{itemize}
      \item Client-server architecture: The \tb{service} consists of server and client parts. \pause
      \item \tb{Statelessness}: Requests and responses are \tb{fully} described by individual REST messages. \pause
        There is no session between client and server (at the \tb{HTTP} communication level, but possibly at other communication levels). \pause
      \item Caching: Because of statelessness, it is (very often) reasonable to cache responses. \pause
      \item \tb{Uniform} interface (fundamental design condition):
        \begin{itemize}
          \item \tb{Resource addressing} through URIs \pause
          \item Resource modification through appropriate resource representations \pause
          \item \tb{Self-descriptive} messages \pause
          \item Hypermedia as the Engine of Application State (\tb{HATEOAS}) \pause
        \end{itemize}
      \item Layered systems: Behind the interface, the service is implemented through the interaction of layered applications and services.
    \end{itemize}
  \end{frame}

\begin{frame}[fragile]{Responses\footnote{Also taken from POOSE and translated.}}
    Structure of a \tb{response}:
    
    \begin{lstlisting}[language=pltiny]
  PROTOCOL_VERSION | STATUS_CODE | STATUS_MESSAGE
  CONTENT (Optional)
    \end{lstlisting}\pause
    
    The valid status codes and status messages are divided into five groups\footnote{A complete overview can be found here, among others: \url{https://http.cat/}}.
    \begin{itemize}
      \item Informational responses (100-199)
      \item \tb{Successful} responses (200-299)\footnote{Fun fact: the unofficial status code 218 \glqq This is fine\grqq\ is used to smuggle errors past a reverse proxy.} Example: \lpl{200 = OK}
      \item Redirection messages (300-399)
      \item \tb{Client} error responses (400-499) Example: \lpl{404 = RESOURCE NOT FOUND}
      \item \tb{Server} error responses (500-599)
    \end{itemize}\pause
    
    The type of the received \lpl{CONTENT} is usually the desired return type.
    
    \note{\LARGE XX:51}
  \end{frame}

	A function can also take only \tb{one parameter} \pp{request: fastapi.Request}\furl{https://fastapi.tiangolo.com/advanced/using-request-directly/}.\newline
	%
	This contains the \tb{entire} context: \pp{path_params}, \pp{query_params}, \pp{cookies}, etc.\newline
	%
	When using the \pp{Request} class, there is \tb{no} automatic type validation anymore.
\end{frame}



\begin{frame}{Defining Routes}
	A \tb{route} can be defined using the \pp{app.get()} or \pp{app.post()} \tb{decorator}.
	
	\medskip
	
	Another option is \pp{app.add_api_route(path, endpoint, methods)}.\newline
	%
	This becomes especially relevant when \tb{routes} are split across \tb{multiple files} or when \tb{methods} should be used as endpoints.
	
	\medskip
	
	For splitting endpoints, FastAPI provides the \tb{APIRouter}\furl{https://fastapi.tiangolo.com/reference/apirouter/} class.\footnote{A real-life example: \url{https://github.com/MayNiklas/whisper_api/blob/pydantic-v2-migration/src/whisper_api/api_endpoints/endpoints.py}}
	
	\medskip
	
	It is good practice to put all routes of an API version into a \tb{namespace}.\newline
	Example: \pp{url:port/api/v1/here_the/route_name}.
	
	\medspace
\end{frame}



\begin{frame}{Responses}
	Every endpoint function must return a \tb{response}.
	
	\medskip
	
	A response\footnote{\url{https://fastapi.tiangolo.com/tutorial/response-model/}} can be:
	
	\medskip
	
	\begin{itemize}
		\pause
		\item a \pp{dict}.
		\pause
		\item an object that inherits from \tb{Pydantic}'s \pp{BaseModel}.
		\pause
		\item a response from \pp{fastapi.responses}, e.g. \pp{JSONResponse}.
		\pause
		\item a raised \pp{fastapi.HTTPException}. %Adrian told me so
		\pause
	\end{itemize}
	
	\medskip
	
	\pp{BaseModel} ensures better \tb{automatically} generated documentation\footnote{This also applies to objects that inherit from BaseModel.}.
	
	\medskip
	
	% TODO: add an example with status code
	
	The \tb{response type}, as well as the expected \tb{status code}, can be \tb{created} and returned normally within the function.
	
	\medskip
	
	Alternatively, the expected types can already be set when \tb{registering} the route.
	
\end{frame}


\begin{frame}[fragile]{FastAPI - Zweites Beispiel}
    \begin{lstlisting}[language=pytiny]
import uuid
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
from pydantic import BaseModel

app = FastAPI()

class User(BaseModel):
    name: str
    id: uuid.UUID | None = None

    def model_post_init(self, ctx):
        if self.id is None:
            self.id = uuid.uuid4()

@app.post("/api/v1/register/{u_name}")
async def read_item(request: Request):
    u = User(name=request.path_params["u_name"])
    # could also be just 'return u' in case of a 200 response
    return JSONResponse(content=u.model_dump_json(), status_code=200)
    \end{lstlisting}
\end{frame}


\begin{frame}[fragile]{FastAPI - A Cool Concept}
	FastAPI provides an \tb{entrypoint} to initialize resources before receiving requests and to release them again after processing the last request.\newline
	This is implemented via the \tb{lifespan}\furl{https://fastapi.tiangolo.com/advanced/events/} function\footnote{Strictly speaking, an asynchronous generator}.\newline
	Initialization and release are separated by the \pp{yield} keyword.
	\begin{lstlisting}[language=pytiny]
@asynccontextmanager
async def lifespan(app: FastAPI):
    # init some things before api starts
    start_some_threads()
    yield
    # kill the things, api has shut down
    join_the_threads()
app = FastAPI(lifespan=lifespan)  # register lifespan function
	\end{lstlisting}
	The first part of the function is executed before startup, the second after shutdown.
	\nsl{Again: the function name is arbitrary.\\}
	\nsl{why exactly do you need this? why not just make everything global? - global vars are bad style, separate initialization from functionality\\}
	
\end{frame}

\section{A Clean Shutdown}
\subsection{atexit}
\begin{frame}[fragile]{atexit}
        A (gracefully) \tb{terminated} program, is able to shut down in a controlled manner.

		\medskip
		
		This \tb{time} can be used to transfer data from memory to a persistent medium, remove temporary files that were created, and more.
		
		\medskip
		
		Python provides the \pp{atexit} module specifically for this purpose.
		
		\medskip
		
		This module offers a \tb{decorator} as well as an explicit \pp{register} function.
		
		\medskip
		
		Functions \tb{registered} here are executed when the program receives the signal to terminate.
		
		\medskip


 \begin{lstlisting}[language=pytiny]
@atexit.register
def goodbye():
    print('You are now leaving the Python sector.')	

def goodbye_with_args(name: str, adjective: str):
   print(f"Goodbye {name}, was {adjective} to meet you!")

atexit.register(goodbye_with_args, 'Donny', 'nice')
	\end{lstlisting}
\end{frame}
