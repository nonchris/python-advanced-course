\subsection{Brief Rundown of AsyncIO}
\begin{frame}{Async: Informal in Four Points + Conclusion}
	\begin{itemize}
		\item \tb{Asynchronous} programs can \tb{switch} between the \tb{execution} of different functions when one function is \tb{blocking} while waiting for (usually) I/O.
		
		\pause
		
		\item[] The entire program still runs in \tb{one} thread\footnote{Nevertheless, the execution of the program is unpredictable, so it may be necessary to protect resources!}.
		
		\pause
		
		\item A \tb{function} marked as \pp{async}, means that it will \tb{wait} for something (usually) I/O.
		
		\pause
		
		\item Such methods are called \tb{coroutines}.
		
		\pause
		
		\item Coroutines \tb{must}, in the context of Discord.py, be directly \tb{awaited} with \pp{await}.
		
		\pause
		
		\item[] \tb{As soon as} the program encounters \pp{await}, it can continue working at \tb{another} place and \tb{return} to this call once the coroutine has completed.
		
		\item \tb{Conclusion}: With \pp{asyncIO}, \tb{no time} is \tb{wasted} waiting for I/O.
		\nll{More later or in apendix}
	\end{itemize}
\end{frame}

\begin{frame}{Agent - Definition}
An \tb{Agent} is an LLM instance prompted to \tb{autonomously complete} a specific {task}.

\medskip

To complete that task, the LLM can \tb{invoke} provided \tb{tools} to gather further information or interact with certain parts of the real world, e.g., a file system.

\medskip

An agent is invoked on an \tb{initial} set of \tb{messages}, providing the base context and goal to achieve.

\medskip

Each LLM and tool \tb{output} is \tb{appended} to the message context, furthering the reasoning of the Agent.
\end{frame}

\begin{frame}{Tools - Definition}
	\tb{Tools} are the Agent's main \tb{interface} for completing the provided task.
	
	\ms
	
	\medskip
	
	The tool's functionality is \tb{implemented} and \tb{executed} on the \tb{application side}.
	The LLM and LLM provider do not receive or execute the tool's code.
	
	\ms
	
	The only information the LLM receives are the \tb{signature} and \tb{description} of the tool, which are provided by the application.
	
	\ms
	
	To use a tool the LLM must \tb{explicitly} generate a call of the tool in its response.\newline
	The LLM response can contain \tb{multiple} tool calls.
	
	\ms
	
	After a tool was executed the Agent requires some kind of \tb{response} from the tool in form of a new message.
	\nsl{You can ob just terminate the agent or delete the last message from the chat...}
	
\nll{Tool Calling must be supported by LLM\\id linking}
\end{frame}

\begin{frame}[fragile]{Tools - Dependency Injection}
	Tools don't live in a vacuum. Many tools require \tb{more context}. This can range from simple state to access to the full message history.
	
	\ms
	
	Pydantic AI has \tb{Dependency Injection} as one of its core features.
	
	\ms
	
	Each tool may receive a \pp{RunContext} as its first \tb{argument} if declared as such. 
	This object provides access to the full context of the agent execution the tool call is a part of. 
	Including the \tb{message history}.
	
	\ms
	
	Additionally it can provide \tb{access} to an arbitrary, \tb{predefined} context object accessible using \pp{.deps}. Its type can be defined on agent definition.
	This object can be used to \tb{carry} further \tb{state} through the agent execution.
	
	\nll{Tools dont live in a vacuum\\We can track state in the tools.\\t is typed and intellisense gets it\\this example is stripped down and not functional}
\end{frame}


\begin{frame}[fragile]{Tools - Dependency Injection}
	\begin{lstlisting}[language=pytiny]
class WikiRaceTask(BaseModel):
    current_pool: list[str] | None
    chosen_words: list[str] = Field(exclude=True, default_factory=list)

# SPECIFY which type of context the tool expects
def choose_word(ctx: RunContext[WikiRaceTask], word: str) -> str:
    if word not in ctx.deps.current_pool:
        return f"The word '{word}' is not in the available pool."

    ctx.deps.current_pool  = WikiRaceTask.get_word_pool(word)
    ctx.deps.chosen_words.append(word)
    return "The available words are: " + " ".join(ctx.deps.current_pool)
    
wiki_agent = Agent("openai:gpt-4o", deps_type=WikiRaceTask,  # SPECIFY agents type of dependency
    output_type=[WikiRaceResult, str], instructions=system_prompt, tools=[Tool(choose_word)])
    
res = wiki_agent.run_sync(
    f"Your target is: '{target}'. Your current pool is: {init.get_formatted_words()}.",
    deps=WikiRaceTask(current_pool=None),  # PASS object instance (type must match set deps_type)
)
	\end{lstlisting}
		\nll{Tools dont live in a vacuum\\We can track state in the tools.\\t is typed and intellisense gets it\\this example is stripped down and not functional}
\end{frame}


\begin{frame}[fragile]{Agent Run Methods}
	An Agent can be run in full \tb{until} it \tb{terminates} using \pp{.run_sync()} and \pp{async .run()}.
	
		\begin{lstlisting}[language=pytiny]
res = wiki_agent.run_sync(
    f"Your target is: '{target}'. Your current pool is: {init.get_formatted_words()}.",
    deps=init,
)
			\end{lstlisting}
	
	\ms
	
	Alternatively it can be run\tb{ step by step} using \pp{.iter()}, which traverses the agents graph node by node.
			\begin{lstlisting}[language=pytiny]
nodes = []
init_prompt = f"Your target is: '{target}'. Your current pool is: {init.get_formatted_words()}."
async with wiki_agent.iter(init_prompt, deps=init) as agent_run:
    async for node in agent_run:
        # Each node represents a step in the agent's execution
        nodes.append(node)
	\end{lstlisting}
The latter makes \tb{state} management more \tb{complex}. E.g. no direct access to all messages at the end.
	\nll{Iter makes it easier to 'interactively' print/log the chat as it advances.\\Else.: Do it in the TCs I guess...}
\end{frame}


\begin{frame}{Agent Termination / Output Functions}
An agent has to terminate with the call to one of the \tb{Output Functions}\furl{https://ai.pydantic.dev/output/} \footnote{Pydantic AI does not distinguish between 'tool calling' and 'function calling'.}.%\newline
% TODO
%Alternatively a \tb{tool} can \tb{return} the required output \tb{type}, thereby terminating the agent.

\ms

If not defined otherwise these \tb{tools} are \tb{generated} from the types defined as \pp{output_type} in the agent initialization.
The tool \tb{descriptions} are fetched from the \tb{docstring} of the class if not defined otherwise.
They can also be defined using the class \pp{ToolOutput}\furl{https://ai.pydantic.dev/api/output/\#pydantic_ai.output.ToolOutput}.

\ms

It is possible to \tb{apply} further output \tb{validators} beside the potential type validation by Pydantic.

\ms

If the output validator classifies the \tb{output} as \tb{insufficient} or in general a tool \tb{call} is \tb{not satisfying} it's possible to raise a \pp{ModelRetry}\furl{https://ai.pydantic.dev/api/exceptions/?query=ModelRetry\#pydantic_ai.exceptions.ModelRetry}, which asks the LLM to try again.
\end{frame}


\begin{frame}[fragile]{Output / Context Types}
On \tb{tool} execution you can opt to \tb{receive} the \pp{RunContext[<YourType>]}\furl{https://ai.pydantic.dev/api/tools/\#pydantic_ai.tools.RunContext} which provides \tb{context} of the current agents execution (messages, other metadata) and - if set - your \tb{injected dependency}.

\ms

After a graceful \tb{termination} of the agent (using the run methods) you receive a\newline \pp{AgentRunResult}\furl{https://ai.pydantic.dev/api/agent/\#pydantic_ai.agent.AgentRunResult} as output.
This object provides you with all \tb{context} of the execution, as well as well as the \tb{output} of the agent (\pp{<AgentRunResult>.output}).
\end{frame}


\begin{frame}[fragile]{Output Mode Considerations}
When \tb{requiring} a \tb{structured} output at the end most LLMs will \tb{stop} to do produce any \tb{text} content\furl{https://github.com/pydantic/pydantic-ai/issues/2158} and only produce tool calls\footnote{This does not apply to reasoining models}.

\ms

If text content is wanted \pp{str} must be also a \tb{valid output} format, resulting in at least two possible return types to account for when handling the agent's result.

\ms

Alternatively it's possible to explicitly provide the LLM with a \tb{Think Tool}\furl{https://www.anthropic.com/engineering/claude-think-tool}.

\ms 

It might be also possible to circumvent this limitation in PydanticAI using an output validator and the \pp{ModelRetry} feature\furl{https://github.com/pydantic/pydantic-ai/issues/2793}.
\end{frame}



\begin{frame}[fragile]{Processing the Message History}
\tb{History Processors} are functions registered in the Agent and called before every request to the LLM.
They allow \tb{processing} and \tb{altering} the history before the call.

\ms

Using processors the agents history can be kept more concise, by simply dropping old messages or more advanced summarization approaches\furl{https://ai.pydantic.dev/message-history/\#summarize-old-messages}.

\ms

Since these processors are integrated in the graph of the agent, they can be used for logging purposes too, e.g. dumping the current conversation in case the next LLM call exceeds the set UsageLimits.

\begin{lstlisting}[language=pytiny]
from pydantic_core import to_jsonable_python
def log_msgs(hist: list[ModelMessage]) -> list[ModelMessage]:
    Path(f"log-file.json").write_text(json.dumps(to_jsonable_python(hist), indent=4))
    return hist
    
Agent(..., history_processors=[log_msgs])
\end{lstlisting}
\end{frame}


\begin{frame}[fragile]{Usage Limits}
	To \tb{safeguard} against excessive LLM calls, rapidly depleeting your wallet, you can \tb{specify} \pp{UsageLimits}\furl{https://ai.pydantic.dev/api/usage/\#pydantic_ai.usage.UsageLimits}.
	
	\ms
	
	Limits can be the number of agent requests, the number of tokens, etc.
	
	\ms
	
	If a \tb{limit} is reached a \pp{UsageLimitExceeded} is \tb{raised}. Discarding the full agent's conversation.
	\nll{Chat has to be logged or smthng to keep it after an exception happend.}
	
	\ms
	
	There might be cases where using \pp{Agent.iter()} is the better option than \pp{.run()}, since it gives more control over the processing of a failed chat. Or logging it using a History Processor.
\end{frame}


\begin{frame}[fragile]{Agent Interaction}
	It's possible to have \tb{multiple agents} do certain sub-tasks.
	
	\ms
	
	New agents can be \tb{started} at all points where you have \tb{access} to the code execution. E.g. in a tool, in the history processors, after an agent has terminated.
	
	\ms
	
	For agents \tb{reacting} to each other the simplest approach is building a \tb{loop} which alternately calls the agents with each other's results.
	
	\nll{Example: Guess series}
\end{frame}

\begin{frame}{Limitations of Agents}
	LLMs are \tb{non-deterministic}, hence you can't rely on getting the same output for the same input.
	
	\pause
	\ms
	
	LLMs have only a \tb{limited context} length.
	Requests to commercial LLMs are billed by used tokens, so larger context sizes result in more expesnive requests.
	
	\pause
	\ms
	
	LLMs are not good for keeping secrets or follow strict orders. With enough \tb{gaslighting} it's 'easily' possible to get the LLM to disobey its initial prompting.
	
	\pause
	\ms
	
	Just because you \textsl{\tb{could}} do it with an agent doesn't mean you \textsl{\tb{should}}.
	\nll{Google translate}
	
\end{frame}

\section{asyncIO}

\begin{frame}{Motivation}
	\tb{IO operations} are by default \tb{synchronous} and therefore blocking.
	\newline
	Simple examples: reading from a file, \pp{time.sleep()}
	
	\medskip
	
	Useful work could be done elsewhere during this \tb{waiting time}.
	
	\medskip
	
	This is especially desirable in programs that should \tb{respond} to \tb{requests} or events in general. 
	
\end{frame}


\begin{frame}{What is asyncIO?}
	\pp{asyncio} is a \tb{built-in} Python library.
	\newline
	It introduces two new \tb{keywords}: \pp{async} \& \pp{await}.
	
	\medskip
	
	It provides an \tb{asynchronous} program loop running in \tb{a single thread}.
	
	\medskip
	
	Using \pp{asyncio} allows the currently executing function to \tb{pause} at certain points, letting another function continue that is not waiting on IO.
	
\end{frame}


\begin{frame}{What does async do?}
	Functions intended to run asynchronously are \tb{marked} with \pp{async def(...):}.
	\newline
	Within these functions, other asynchronous functions can be called and awaited.
	
	\medskip
	
	Calling an \pp{async} function returns a \tb{Coroutine} object instead of executing the function immediately. 
	
	\medskip
	
	The keyword \pp{await} \tb{blocks} the program flow of the function until the \pp{awaitable}\footnote{Async tasks are also awaitable: \url{https://docs.python.org/3/library/asyncio-task.html\#awaitables}} completes. The result is then returned.
	
	\medskip
	
	Async-\tb{tasks} are interesting because they allow multiple coroutines to run \tb{quasi-concurrently}.
	
\end{frame}



\begin{frame}{Awaitable}
	
	An \pp{awaitable}\furl{https://docs.python.org/3/library/asyncio-task.html\#awaitables} is defined by implementing \pp{\_\_await\_\_()}\footnote{There are also \_\_aiter\_\_(), \_\_aenter\_\_() and more}. \newline
	It can therefore be awaited using \pp{await}.
	
	\medskip
	
	The three built-in types of \pp{awaitables} are \tb{Coroutine}, \tb{Task} and \tb{Future}\footnote{Not discussed further here.}.
	
\end{frame}



\begin{frame}[fragile]{Starting an Async Loop}
	The asynchronous program flow is started by \tb{calling} \pp{asyncio.run()}.
	
	\medskip
	
	The function takes a \tb{Coroutine} as argument.
	\begin{lstlisting}[language=pytiny]
		asyncio.run(main())
	\end{lstlisting}
	$\rightarrow$ The \tb{passed} function is unintuitively \tb{"executed"} when passed.
	
	\medskip
	
	The \tb{start} of the loop is \tb{blocking}.
	
\end{frame}


\begin{frame}{Async-Coroutine}
	A \tb{Coroutine} object\furl{https://docs.python.org/3/library/asyncio-task.html\#coroutines} is \tb{created} by \tb{"calling"} an \pp{async} function.
	
	\medskip
	
	The actual function is \tb{not executed} until it is \tb{passed to a Task} or \tb{awaited}.
	
	\medskip
	
	A \pp{coroutine} must be awaited before the end of the creating function.
	
	\medskip
	
	It is not possible to return an un-awaited \pp{coroutine}.
\end{frame}

\begin{frame}{Async-Tasks}
	An async-\tb{Task}\furl{https://docs.python.org/3/library/asyncio-task.html\#task-object} is scheduled for execution using \pp{asyncio.create_task()}.
	\newline
	This allows \tb{concurrent behavior}.
	
	\medskip
	
	The completion and \tb{result} can be \pp{awaited}.
	\newline 
	The \tb{task state} can be checked using \pp{.done()}.
	\newline
	The result of a completed task can be retrieved with \pp{.result()}.
	
	\medskip
	
	\pp{asyncio.gather()} waits for all passed tasks to finish and returns the results in the same order as passed.
	
	\medskip
	
	A task does not necessarily have to be awaited.
	
\end{frame}

\begin{frame}[fragile]{Async-Tasks Example}
	\begin{lstlisting}[language=pytiny]
import asyncio

async def sleep(name: str, st=1):
    """ helper to execute an 'IO-heavy' task (sleeping) """
    print(f"[{name}] Before await")
    await asyncio.sleep(st)
    print(f"[{name}] After await")
    return name

async def main():
    task1 = asyncio.create_task(sleep("call1"))  # create task
    task2 = asyncio.create_task(sleep("call2"))
    print(type(task1))  # >> <class '_asyncio.Task'>

    res1, res2 = await asyncio.gather(task1, task2)  # wait for tasks to finish
    print(res1, res2)  # >> call1, call2

# start main async-loop
asyncio.run(main())
	\end{lstlisting}
	
\end{frame}


\begin{frame}[fragile]{Async-Taskgroup}
	Same example as above using a \tb{TaskGroup}\furl{https://docs.python.org/3/library/asyncio-task.html\#task-groups}.
	
	\medskip
	
	Multiple tasks can be easily scheduled in it.
	
	\medskip
	
	Leaving the \tb{context block} ensures all tasks are \tb{completed}.
	
	\begin{lstlisting}[language=pytiny]
async def main():
    print(f"started at {time.strftime('%X')}")  # %X -> hh:mm:ss
    async with asyncio.TaskGroup() as tg:
        task1 = tg.create_task(sleep('call1'))

        task2 = tg.create_task(sleep("call2"))

    print(f"ended at {time.strftime('%X')}")

    print(task1.result())

asyncio.run(main())
	\end{lstlisting}
	
\end{frame}


\begin{frame}{Where can Async functions be interrupted?}
	
	In short: Anywhere an \tb{interaction} with \pp{asyncio} happens.
	
	\medskip
	
	Program flow can pause when calling \pp{async} functions and using \pp{await}, and continue elsewhere.
	
\end{frame}



\begin{frame}{Protecting Resources}
	When multiple functions can run \tb{'simultaneously'}, shared \tb{resources} may need to be \tb{protected}\footnote{\url{https://docs.python.org/3/library/asyncio-sync.html\#synchronization-primitives}}.
	
	\medskip
	
	Unlike multithreading, not all accesses need protection.
	
	\medskip
	
	A resource should be \tb{protected} if its consistency is required across multiple \tb{async calls}.
	
	\medskip
	
	Also, libraries may \tb{'internally'} use \pp{asyncio}, transferring control back to the library.
	
\end{frame}


\begin{frame}[fragile]{Timeout}
	With \pp{asyncio.timeout()}\footnote{\url{https://docs.python.org/3/library/asyncio-task.html\#timeouts}} a \pp{Task} can be terminated after a set duration.
	
	\medskip
	
	Timeouts can be set and rescheduled with \pp{asyncio.Timeout}.
	\newline
	It can also be queried whether the timeout was reached.
	
	\begin{lstlisting}[language=pytiny]
async def main():
    try:
        async with asyncio.timeout(10):
        await long_running_task()
    except TimeoutError:
        print("The long operation timed out, but we've handled it.")
	\end{lstlisting}
	
\end{frame}



\begin{frame}[fragile]{Async-Subprocess}
	\pp{asyncio} provides an interface to run subprocesses\footnote{Better known as shell commands}.
	
	\medskip
	
	More information here:\newline \url{https://docs.python.org/3/library/asyncio-subprocess.html}
	
\end{frame}
